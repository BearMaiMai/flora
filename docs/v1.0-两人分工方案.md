## 2. 分工策略与原则

### 2.1 为什么现在适合分工？

| 维度 | 评估 |
|------|------|
| **代码耦合度** | ✅ 低。UI 样式（`.wxss` / `.wxml`）与后端对接（`.js` 逻辑 + 云函数）改动文件几乎不重叠 |
| **协作机制** | ✅ 完善。分支策略、PR 规范、Review 流程已全部建好 |
| **任务边界** | ✅ 清晰。前端 UI 和后端数据是两条独立的线，互不阻塞 |
| **并行效率** | ✅ 高。两条线同时推进，合并后即可得到"好看 + 能用"的版本 |

### 2.2 分工原则

```
                    ┌─────────────────────────────────┐
                    │        v1.0 两人分工策略          │
                    │                                  │
                    │    按 「前端 UI」+「后端对接」     │
                    │    两条线并行推进                  │
                    └──────────┬──────────┬────────────┘
                               │          │
                    ┌──────────▼──┐  ┌────▼───────────┐
                    │  Person A   │  │   Person B     │
                    │  前端 UI    │  │   后端 + 数据   │
                    │  精美化     │  │   对接          │
                    │             │  │                 │
                    │ 改: .wxss   │  │ 改: .js (逻辑)  │
                    │ 改: .wxml   │  │ 改: 云函数       │
                    │ (结构微调)  │  │ 新增: service层  │
                    │ 新增: 图片  │  │ 新增: 数据库     │
                    └──────────┬──┘  └────┬───────────┘
                               │          │
                    ┌──────────▼──────────▼────────────┐
                    │        集成到 dev 分支             │
                    │     好看 + 能用 的完整版本         │
                    └─────────────────────────────────┘
```

### 2.3 冲突最小化规则

为了让两人并行开发时**几乎零冲突**，双方需遵守以下约定：

| 规则 | Person A（UI） | Person B（数据） |
|------|---------------|-----------------|
| `.wxss` 文件 | ✅ 主要修改 | ❌ 不修改 |
| `.wxml` 文件 | ✅ 可修改（加类名、调结构） | ⚠️ 尽量不改，如需改则提前沟通 |
| `.js` 文件 - `data` 部分 | ❌ 不修改 | ✅ 主要修改（替换 Mock 数据） |
| `.js` 文件 - 方法部分 | ⚠️ 仅添加动画/交互方法 | ✅ 主要修改（接入 service 调用） |
| 云函数目录 | ❌ 不涉及 | ✅ 主要修改 |
| `utils/` 目录 | ❌ 不涉及 | ✅ 新增 service 层 |
| `styles/` 目录 | ✅ 可修改全局样式 | ❌ 不涉及 |
| `images/` 目录 | ✅ 新增图片资源 | ❌ 不涉及 |
| `app.js` | ❌ 不涉及 | ✅ 配置云环境 |
| `app.wxss` | ✅ 可修改全局样式 | ❌ 不涉及 |

#### Service 层设计参考

```javascript
// miniprogram/utils/service.js

/**
 * 统一云函数调用封装
 * 所有页面通过 service 层调用云函数，不直接使用 wx.cloud.callFunction
 */

const call = (name, action, data = {}) => {
  return wx.cloud.callFunction({
    name,
    data: { action, ...data },
  }).then(res => {
    if (res.result.code === 0) return res.result.data
    throw new Error(res.result.message || '请求失败')
  })
}

module.exports = {
  // 花卉百科
  flower: {
    list: (params) => call('flower', 'list', params),
    detail: (id) => call('flower', 'detail', { id }),
    search: (keyword) => call('flower', 'search', { keyword }),
    recommend: () => call('flower', 'recommend'),
  },
  // 用户
  user: {
    login: (userInfo) => call('user', 'login', { userInfo }),
    updateInfo: (data) => call('user', 'updateInfo', data),
    toggleFavorite: (flowerId) => call('user', 'toggleFavorite', { flowerId }),
    getFavorites: () => call('user', 'getFavorites'),
    getStats: () => call('user', 'getStats'),
  },
  // 我的植物
  plant: {
    add: (data) => call('plant', 'add', data),
    list: () => call('plant', 'list'),
    update: (id, data) => call('plant', 'update', { id, ...data }),
    remove: (id) => call('plant', 'remove', { id }),
  },
  // 成长日记
  diary: {
    add: (data) => call('diary', 'add', data),
    list: (plantId) => call('diary', 'list', { plantId }),
    delete: (id) => call('diary', 'delete', { id }),
  },
  // 养护提醒
  reminder: {
    list: () => call('reminder', 'list'),
    complete: (id) => call('reminder', 'complete', { id }),
  },
  // 公共
  common: {
    getHomeData: () => call('common', 'getHomeData'),
    getDailyTip: () => call('common', 'getDailyTip'),
  },
}
```

```
        ├── feature/ui-polish          ← Person A 工作分支
        │
        └── feature/cloud-bindling     ← Person B 工作分支
```

### 5.2 初始化步骤

```bash
# ── 两人都需要执行 ──

# 1. 拉取最新代码
git pull origin preview

# 2. 从 preview 创建 dev 分支（仅需一次，由 Admin 执行）
git checkout preview
git checkout -b dev
git push origin dev

# ── Person A ──
git checkout dev
git checkout -b feature/ui-polish

# ── Person B ──
git checkout dev
git checkout -b feature/cloud-bindling
```

### 5.3 日常协作流程

```
  Person A                                Person B
    │                                       │
    ├── 在 feature/ui-polish 上开发          ├── 在 feature/cloud-bindling 上开发
    ├── 每完成一个页面就 commit               ├── 每完成一个模块就 commit
    │                                       │
    │   ┌───── 每 2-3 天集成一次 ──────┐     │
    │   │                              │     │
    ├── push → PR → dev ◄── PR ← push ──────┤
    │   │      ↓                       │     │
    │   │   解决冲突（如有）             │     │
    │   │      ↓                       │     │
    │   │   合并到 dev                  │     │
    │   └──────────────────────────────┘     │
    │                                       │
    ├── git merge dev （同步最新 dev）        ├── git merge dev
    │                                       │
    │   ┌───── 阶段完成后 ─────────────┐     │
    │   │   dev → preview (PR + 审核)  │     │
    │   │   preview → main (发布)      │     │
    │   └──────────────────────────────┘     │
```

### 5.4 Commit Message 规范

两人统一使用以下前缀：

| 前缀 | Person A 使用场景 | Person B 使用场景 |
|------|------------------|------------------|
| `feat(ui):` | 新增/优化页面样式 | — |
| `feat(service):` | — | Service 层新增 |
| `feat(cloud):` | — | 云函数修改 |
| `feat(data):` | — | 数据库/数据导入 |
| `fix(ui):` | 修复样式问题 | — |
| `fix(cloud):` | — | 修复云函数问题 |
| `chore:` | 图片资源变更 | 配置变更 |
| `style:` | 纯样式调整（无功能变化） | — |

## 7. 风险与注意事项

### 7.1 潜在风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 花卉数据编写量大（50种 × 20+字段） | Person B 进度延迟 | 🔴 高 | 第一轮先做 20 种核心花卉，后续分批补充 |
| `.wxml` 结构被同时修改导致冲突 | 合并困难 | 🟡 中 | 严格遵守分工规则，A 改结构前告知 B |
| 云环境配额不足 | 功能受限 | 🟢 低 | 免费额度初期足够，关注存储和调用次数 |
| 花卉图片版权问题 | 法律风险 | 🟡 中 | 使用 Unsplash/Pexels 免费商用图片 |
| `wx.getUserProfile` 废弃 | 登录流程不可用 | 🔴 高 | Person B 优先调研并适配新隐私接口 |

### 7.2 沟通机制

| 场景 | 沟通方式 |
|------|----------|
| 日常进度 | 每天下班前在群里简要同步今日完成的任务 |
| 需要对方配合 | 在 GitHub Issue / PR 评论中 @对方 |
| 发现冲突 | 立即沟通，约定由谁来修改 |
| 集成日 | 两人同时在线，先合并 → 解决冲突 → 验证功能 |

### 7.3 交接约定

当两人代码需要衔接时（例如 Person A 的 UI 需要 Person B 提供的数据字段），遵循以下约定：

| 约定 | 说明 |
|------|------|
| **数据字段不变** | Person B 对接真实数据时，保持与当前 Mock 数据**相同的字段名和结构** |
| **新增字段通知** | 如果 B 需要在 data 中新增字段，提前告知 A 以便 wxml 中绑定 |
| **图片字段统一** | 花卉图片统一使用 `coverImage` 字段，类型为 String（云存储 fileID 或 HTTP URL） |
| **加载状态统一** | 统一使用 `loading: true/false` 控制骨架屏显示 |
