## 1. 分支策略与保护规则

### 1.1 分支一览

```
main (主分支)              ← 🔒 保护分支，正式发布版本
  │
  ├── preview (预览分支)    ← 🔒 保护分支，体验版 / 预发布
  │
  ├── dev (开发主分支)      ← 日常集成，相对宽松
  │     │
  │     ├── feature/xxx     ← 功能开发分支
  │     ├── fix/xxx         ← Bug 修复分支
  │     └── ...
  │
  └── hotfix/xxx            ← 紧急修复（从 main 切出）
```

### 1.2 🔒 保护分支规则

以下两个分支为**保护分支**，已在 GitHub 仓库配置 Branch Ruleset：

| 分支 | 用途 | 保护规则 |
|------|------|----------|
| **`main`** | 正式发布版本，对应线上环境 | 🚫 禁止直接 push<br>🚫 禁止强制推送（force push）<br>🚫 禁止删除<br>✅ 仅接受 Pull Request 合并<br>✅ 至少 1 人审批通过 |
| **`preview`** | 体验版 / 预发布版本 | 🚫 禁止直接 push<br>🚫 禁止强制推送（force push）<br>🚫 禁止删除<br>✅ 仅接受 Pull Request 合并<br>✅ 至少 1 人审批通过 |

### 1.3 谁可以绕过保护规则？

| 角色 | 能否直接 push 保护分支 | 说明 |
|------|----------------------|------|
| **仓库管理员（Admin）** | ✅ 可以 | 仓库创建者 @BearMaiMai 拥有 Admin 权限，可绕过分支保护规则直接推送 |
| **协作者（Collaborator / Write）** | ❌ 不可以 | 必须通过 Pull Request 提交代码，并经过管理员审核批准后合并 |

> ⚠️ **重要提醒**：即使管理员可以绕过保护规则，也**建议**通过 PR 流程来合并代码，以保持提交历史的可追溯性和规范性。仅在紧急修复等特殊情况下才直接推送。

### 1.4 各分支详细说明

#### `main` — 主分支（生产环境）

- 始终保持**可发布**状态
- 仅通过以下方式接收代码：
  - `preview` → `main` 的 PR（正常发布流程）
  - `hotfix/*` → `main` 的 PR（紧急修复）
  - Admin 直接推送（仅紧急情况）
- 每次合并到 main 后应打 **Git Tag** 标记版本号

#### `preview` — 预览分支（体验版环境）

- 对应微信小程序的**体验版**
- 仅通过以下方式接收代码：
  - `dev` → `preview` 的 PR（功能集成）
  - Admin 直接推送（仅紧急情况）
- 在此分支部署云函数到**测试环境**进行验证

#### `dev` — 开发分支

- 日常开发集成分支
- 协作者可直接 push（无分支保护）
- 功能分支完成后合并到 `dev`
- 定期将 `dev` 通过 PR 合并到 `preview` 进行测试

#### `feature/*` — 功能分支

- 从 `dev` 切出
- 命名格式：`feature/<功能描述>`
- 示例：`feature/flower-list`、`feature/garden-diary`
- 完成后合并回 `dev`

#### `fix/*` — 修复分支

- 从 `dev` 切出（一般 bug）
- 命名格式：`fix/<问题描述>`
- 示例：`fix/reminder-time-bug`

#### `hotfix/*` — 紧急修复分支

- 从 `main` 切出（线上紧急 bug）
- 命名格式：`hotfix/<问题描述>`
- 修复后同时合并到 `main` 和 `dev`

## 3. 开发者工作流程

### 3.1 首次参与项目

```bash
# 1. 克隆仓库
git clone https://github.com/BearMaiMai/flora.git
cd flora

# 2. 切换到 dev 分支
git checkout dev

# 3. 安装云函数依赖
cd cloudfunctions/flower && npm install && cd ../..
cd cloudfunctions/user && npm install && cd ../..
cd cloudfunctions/plant && npm install && cd ../..
cd cloudfunctions/diary && npm install && cd ../..
cd cloudfunctions/reminder && npm install && cd ../..
cd cloudfunctions/common && npm install && cd ../..

# 4. 用微信开发者工具打开项目根目录
```

### 3.2 日常开发流程（协作者）

```
 ┌─────────────────────────────────────────────────────┐
 │                   协作者开发流程                       │
 │                                                      │
 │  ① 从 dev 创建功能分支                                │
 │     git checkout dev                                 │
 │     git pull origin dev                              │
 │     git checkout -b feature/xxx                      │
 │                                                      │
 │  ② 在功能分支上开发、提交                              │
 │     git add .                                        │
 │     git commit -m "feat(xxx): 描述"                   │
 │                                                      │
 │  ③ 推送功能分支到远程                                  │
 │     git push origin feature/xxx                      │
 │                                                      │
 │  ④ 在 GitHub 上创建 Pull Request                      │
 │     feature/xxx → dev （日常合并）                     │
 │     dev → preview    （提测合并，需 Admin 审批）        │
 │     preview → main   （正式发布，需 Admin 审批）        │
 │                                                      │
 │  ⑤ 等待 Code Review + Admin 审批                      │
 │                                                      │
 │  ⑥ 审批通过后由 Admin 合并 PR                          │
 │                                                      │
 │  ⑦ 删除已合并的功能分支                                │
 │     git branch -d feature/xxx                        │
 │     git push origin --delete feature/xxx             │
 └─────────────────────────────────────────────────────┘
```

### 3.3 具体步骤详解

#### Step 1：创建功能分支

```bash
# 确保 dev 是最新的
git checkout dev
git pull origin dev

# 创建功能分支
git checkout -b feature/flower-search
```

#### Step 2：开发 & 提交

```bash
# 开发完成后，暂存修改
git add .

# 提交（遵循 Commit Message 规范，详见开发规范文档第 10 节）
git commit -m "feat(flower): 添加花卉搜索功能

- 新增搜索栏组件
- 实现关键词模糊搜索
- 支持搜索历史记录

--story=0"
```

#### Step 3：推送到远程

```bash
git push origin feature/flower-search
```

#### Step 4：创建 Pull Request

1. 打开 GitHub 仓库页面
2. 点击 **"Compare & pull request"** 按钮（推送后会自动出现）
3. 填写 PR 信息（参见第 4 节 PR 规范）
4. 选择目标分支：
   - 功能开发完成 → 目标选 `dev`
   - 准备提测 → 目标选 `preview`（需 Admin 审批）
   - 正式发布 → 目标选 `main`（需 Admin 审批）
5. 点击 **"Create pull request"**

#### Step 5：等待审核

- Admin 会收到 PR 通知
- 如有修改意见，在本地修改后再次 push 到同一分支即可自动更新 PR
- 审核通过后，Admin 会合并 PR

## 4. Pull Request 规范

### 4.1 PR 标题格式

```
<type>(<scope>): <简要描述>
```

**示例：**
- `feat(flower): 添加花卉搜索功能`
- `fix(reminder): 修复提醒时间显示错误`
- `docs: 更新 README 使用说明`

### 4.2 PR 描述模板

创建 PR 时，请按以下模板填写描述：

```markdown
## 📋 变更说明

简要说明本次 PR 做了什么。

## 🔗 关联需求

- 关联 PRD 用户故事：US-xxx
- 关联 Issue：#xxx（如有）

## 📝 变更内容

- [ ] 新增了 xxx 功能
- [ ] 修改了 xxx 逻辑
- [ ] 修复了 xxx 问题

## 📸 截图/录屏（如涉及 UI 变更）

| 修改前 | 修改后 |
|--------|--------|
| 截图   | 截图   |

## ✅ 自检清单

- [ ] 代码已在微信开发者工具中通过编译
- [ ] 已在模拟器/真机上测试核心流程
- [ ] 无多余的 console.log / debugger
- [ ] 命名符合开发规范
- [ ] 新增页面已在 app.json 中注册
- [ ] 云函数修改已在开发环境测试通过
```

### 4.3 PR 合并规则

| 目标分支 | 来源分支 | 审批要求 | 合并方式 |
|---------|---------|---------|---------|
| `dev` | `feature/*`, `fix/*` | 建议但非强制 | Squash and merge（推荐） |
| `preview` | `dev` | **必须 Admin 审批** | Create a merge commit |
| `main` | `preview`, `hotfix/*` | **必须 Admin 审批** | Create a merge commit |

## 5. Code Review 规范

### 5.1 审核重点

| 检查维度 | 关注点 |
|---------|--------|
| **功能正确性** | 是否实现了需求描述的功能 |
| **代码规范** | 命名、格式、注释是否符合[开发规范文档](./开发规范文档.md) |
| **安全性** | 云函数是否校验了 openid、参数是否做了校验 |
| **性能** | setData 是否合并调用、图片是否懒加载、列表是否分页 |
| **兼容性** | 是否考虑了 iOS / Android 差异 |
| **边界处理** | 空数据、网络异常、重复点击等异常情况 |

### 5.2 审核流程

```
提交 PR → Reviewer 审核代码 → 给出意见
                                │
                    ┌───────────┼───────────┐
                    ▼           ▼           ▼
                 Approve    Comment     Request
                 (通过)    (仅评论)     Changes
                    │                   (要求修改)
                    ▼                      │
              Admin 合并 PR          开发者修改后
                                    重新请求 Review
```

### 5.3 Review 评论规范

请在评论中使用以下前缀，明确意见类型：

| 前缀 | 含义 | 是否阻塞合并 |
|------|------|-------------|
| `[必须]` | 必须修改才能合并 | ✅ 阻塞 |
| `[建议]` | 建议修改，但不阻塞 | ❌ 不阻塞 |
| `[提问]` | 不理解，需要解释 | ❌ 不阻塞 |
| `[赞]` | 写得好，点赞 👍 | ❌ 不阻塞 |

**示例：**
```
[必须] 这里缺少 try/catch 错误处理，云函数调用失败会导致页面白屏。

[建议] 这个变量名 `d` 含义不清，建议改为 `difficultyLevel`。

[提问] 这里为什么不用 Promise.all 并行请求？有顺序依赖吗？
```

## 6. 常见问题 FAQ

### Q1：我直接 push 到 main/preview 被拒绝了怎么办？

**A**：这是正常的！保护分支不允许直接 push。请按以下步骤操作：

```bash
# 1. 如果你已经在 main 上做了提交，先创建一个新分支保存
git checkout -b feature/my-changes

# 2. 推送新分支
git push origin feature/my-changes

# 3. 到 GitHub 上创建 PR，目标分支选 main 或 preview

# 4. 把 main 恢复到远程状态
git checkout main
git reset --hard origin/main
```

### Q2：PR 提交后发现有问题需要修改怎么办？

**A**：直接在同一个功能分支上修改并 push，PR 会自动更新：

```bash
# 在功能分支上修改
git checkout feature/my-feature
# ... 修改代码 ...
git add .
git commit -m "fix: 根据 review 意见修改"
git push origin feature/my-feature
# PR 会自动包含新的提交
```

### Q3：如何同步最新的 dev 分支代码？

**A**：

```bash
# 方式一：merge（推荐，保留完整历史）
git checkout feature/my-feature
git merge dev

# 方式二：rebase（历史更干净，但需要 force push）
git checkout feature/my-feature
git rebase dev
git push --force-with-lease origin feature/my-feature
```

### Q4：合并冲突怎么解决？

**A**：

```bash
# 1. 先拉取最新的目标分支
git checkout dev
git pull origin dev

# 2. 切回功能分支，合并 dev
git checkout feature/my-feature
git merge dev

# 3. 解决冲突（编辑器中处理冲突标记 <<<< ==== >>>>）
# 4. 标记冲突已解决
git add .
git commit -m "merge: 解决与 dev 的合并冲突"
git push origin feature/my-feature
```

### Q5：Admin 什么时候适合直接 push 保护分支？

**A**：仅在以下紧急情况下可以绕过 PR 流程：

- 🔥 线上严重 Bug 需要立即修复（如白屏、数据丢失）
- 📝 纯文档修改（如更新 README、修复文档 typo）
- ⚙️ 项目配置紧急调整（如 project.config.json）

> 即使是紧急情况，事后也建议补充说明（在 commit message 中注明原因）。
